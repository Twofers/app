# QR campaign tracking

## Scope

Physical Twofer marketing materials use `https://www.twoferapp.com/r/<slug>`.
There is one generic redirect route, not a public webpage for each business.
The route records a privacy-limited scan for an active campaign and redirects
to the correct configured destination.

## Local implementation

- Migration: `20260815130000_qr_campaign_tracking.sql`
- Public function: `qr-campaign-redirect`
- Admin function: `admin-qr-campaigns`
- Admin page: `/admin/qr-campaigns`
- Website route: `/r/:slug` is rewritten to the public Edge Function.

`qr_campaigns` and `qr_scan_events` are RLS-enabled and have no browser-role
policies or grants. The public Edge Function derives the campaign, business,
and source from the active slug; a browser never supplies those identifiers.

## Destination configuration

Before deployment, configure these Edge Function environment values through
the approved hosted-secret/configuration flow. Do not print their values in
chat, terminal output, commits, or documentation.

- `TWOFER_IOS_APP_STORE_URL`
- `TWOFER_ANDROID_PLAY_STORE_URL`
- `SITE_URL`
- `QR_SCAN_IP_HASH_SECRET`

The current website store-link configuration does not yet contain confirmed
store destinations. Do not enable an `app_download` campaign in production
until the App Store and Google Play URLs are confirmed. If a store URL is
missing or invalid, the redirect safely falls back to the Twofer website.

## Privacy

Each event stores campaign/business/source, timestamp, a coarse device class,
and the redirect target. It does not store an authenticated user ID, email,
cookie, referrer, precise location, claim code, redemption code, or raw IP.

User agents are truncated to 512 characters. When a trusted client IP is
available, the Edge Function stores only a daily HMAC-SHA-256 value; the raw
IP is never written or logged. A daily pg_cron job redacts user-agent and
IP-hash metadata after 30 days, preserving aggregate event counts.

Hashed IPs and detailed user agents are pseudonymous telemetry. Review the
public privacy disclosure before production enablement. If a stricter policy
requires no pseudonymous online identifiers, leave `QR_SCAN_IP_HASH_SECRET`
unset; events then store no IP hash.

## Admin behavior

The admin page requires the existing active-admin session, role permissions,
and MFA policy. `qr.read` controls analytics and QR display; `qr.manage`
controls creation and disabling. Creation generates an opaque slug server-side.
Campaigns are disabled rather than deleted so printed materials cannot be
silently reassigned to a new destination.

The page shows total requests, likely physical scans, likely automated
requests, scans by business, source/material, campaign, and day. QR SVGs are
generated by the authenticated admin Edge Function using a pinned local
encoder; no third-party QR image service receives tracking URLs.

## Validation

Run locally:

```text
npm run typecheck
npm run lint
npm test
npm run typecheck:functions
npm run check:website-supabase
npm run check:website-ui
npm run test:db
npm run release:state
npm run gate:release-state
```

After an explicitly approved development/staging migration, run:

```text
node scripts/probe-rls-smoke.mjs
```

Staging smoke cases:

1. Create a test campaign and verify the generated SVG decodes to its `/r/` URL.
2. Request the route with iPhone, Android, desktop, and bot user agents.
3. Verify valid GET requests create one appropriately classified event.
4. Verify `HEAD`, malformed, and disabled slugs create no event.
5. Verify anon/authenticated PostgREST access to both tables is denied.
6. Verify read-only admins cannot create/disable and MFA-required admins need AAL2.
7. Verify no raw IP, HMAC secret, QR token, or store configuration value appears in logs or audit data.
8. Verify the route keeps redirecting if telemetry insertion is deliberately made unavailable.

## Deployment gates

Applying the migration, deploying either Edge Function, configuring hosted
values, and deploying `website/` are production-changing actions. Each needs
Dan's explicit approval. No mobile release build is needed for this web/admin
feature.
