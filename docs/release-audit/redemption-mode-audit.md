# Redemption Mode Merge Audit — `audit/redemption-mode-merge` @ `680d67f`

**Date:** 2026-06-10
**Scope:** Codex Redemption Mode feature merged into the batch stack (merge parents `d30bd02` batch stack, `938091d` codex branch). Migration `20260712120000_redemption_mode_staff_sessions.sql`, five new edge functions, shared guard/crypto modules, client locked mode, guard edits to 24 existing functions, and the PostgREST surface reachable by a staff token.
**Method:** every claim below was verified against the code on this branch, not against `redemption-mode-feature.md`. Nothing was modified.

---

## Findings (severity-sorted)

### HIGH

**1. The locked counter device holds the owner's full session (access + refresh token) for the entire locked period, and the app now has no way to remotely revoke it.**
[lib/redemption-mode.ts:239](lib/redemption-mode.ts:239) — `activateRedemptionMode()` writes the live owner session (`JSON.stringify(ownerSession)`, including the refresh token) to SecureStore under `OWNER_SESSION_BACKUP_KEY` *before* swapping to the staff session, and it stays there until exit ([lib/redemption-mode.ts:315-325](lib/redemption-mode.ts:315)). The audit plan's design requirement was "the locked device must hold NO owner session material." It does — deliberately, so exit can restore the owner without re-login.
Compounding it, the codex branch converted every in-app sign-out to `scope: "local"` ([lib/auth-app-sign-out.ts:31](lib/auth-app-sign-out.ts:31), [app/(tabs)/settings.tsx:345](app/(tabs)/settings.tsx:345), [components/providers/auth-session-provider.tsx:29,45](components/providers/auth-session-provider.tsx:29)), so the owner can no longer revoke the lost device's refresh token from another device through the app at all. The remove-device warning copy honestly admits this ([components/redemption-mode-settings.tsx:257-260](components/redemption-mode-settings.tsx:257): "change your account password too, because this app cannot remotely erase any owner session saved on that device") — but the severing then depends entirely on Supabase revoking other sessions on password change, which is a GoTrue project behavior nobody has verified for this project. `manage-redemption-devices` `remove` deletes the *staff* user ([supabase/functions/manage-redemption-devices/index.ts:127](supabase/functions/manage-redemption-devices/index.ts:127)) but does nothing about the owner session sitting on the same device.
**Risk:** a lost/stolen counter device, if the platform keystore is compromised (rooted device, no passcode), yields a full owner session — deals, PII, billing, account deletion. Mitigation today is SecureStore (hardware-backed) + the password-change advice.
**Fix direction:** at minimum verify password change revokes refresh tokens in this Supabase project and document it in the lost-device runbook; better, add `auth.admin.signOut(ownerId)`-style revocation or stop persisting the owner session and require re-login on exit.

**2. `owner-redemption-security` `enable` overwrites an existing PIN without requiring the current PIN, defeating the `change`/`disable` protections and the lockout.**
[supabase/functions/owner-redemption-security/index.ts:161-189](supabase/functions/owner-redemption-security/index.ts:161) — the `enable` branch never checks whether a PIN is already enabled. `change` requires the current PIN ([index.ts:221](supabase/functions/owner-redemption-security/index.ts:221)) and `disable` requires it ([index.ts:243-251](supabase/functions/owner-redemption-security/index.ts:243)), but anyone holding the owner's signed-in device can simply call `enable` with a *new* PIN: it replaces `pin_hash` and resets `pin_failed_attempts`/`pin_locked_until` to zero. The Redeem-tab PIN exists precisely to stop a person who has the owner's unlocked phone from redeeming; this hole removes that guarantee, and also lets a brute-forcer clear the 5-attempt lockout. The client UI only shows the enable form when disabled, but the endpoint accepts the call regardless.
**Fix direction:** in the `enable` branch, if `row?.enabled && row.pin_hash`, require and verify the current PIN exactly like `change`.

**3. The migration revokes new-function EXECUTE only `FROM PUBLIC`; per this project's prod-verified standard (commit `fd1e98e`), anon retains EXECUTE on all seven new functions via Supabase default privileges.**
[supabase/migrations/20260712120000_redemption_mode_staff_sessions.sql:189-198](supabase/migrations/20260712120000_redemption_mode_staff_sessions.sql:189) and [:609-612](supabase/migrations/20260712120000_redemption_mode_staff_sessions.sql:609) use `REVOKE ... FROM PUBLIC` then `GRANT ... TO authenticated[, service_role]` — the exact pattern that was proven insufficient live on 2026-06-10 for `purge_user_data` and `deal_claim_counts` (anon stayed callable until explicit `FROM anon` revokes were run in the SQL editor). The same applies to the re-`CREATE OR REPLACE`d `validate_business_invite` / `report_business` / `report_user` at [:739-744](supabase/migrations/20260712120000_redemption_mode_staff_sessions.sql:739) (OR REPLACE preserves the existing ACL, so whatever anon grant exists in prod today survives).
**Exploitability is low** — `preview_staff_redemption`/`confirm_staff_redemption` self-guard via `is_active_redeemer_for_business()` and return `unauthorized` for anon, and the four helpers only echo JWT claims — but the standard exists because grants must match intent, and anon being able to *invoke* the redemption RPCs at all is wrong. **The exact post-apply revoke block is in the summary below.** This is a deploy-time blocker, not a code rewrite.

### MEDIUM

**4. `confirm_staff_redemption` / `preview_staff_redemption` have no brute-force lockout and never write `failed_redeem_attempts` — the staff path bypasses the Batch 6 protection entirely.**
`redeem-token` rate-limits to 10 failed attempts per 5 minutes per business/IP and records every failure ([supabase/functions/redeem-token/index.ts:91-131](supabase/functions/redeem-token/index.ts:91)). The staff RPCs ([migration :327-440, :442-607](supabase/migrations/20260712120000_redemption_mode_staff_sessions.sql:327)) accept unlimited guesses of 4–8 char short codes with no counter, no delay, and no audit of failures (the `redemptions` table only records successes). A stolen-but-still-active staff token can grind short codes for its business at HTTP speed. Codes are ~36⁶ so blind brute force is slow, but the asymmetry with the owner path is a regression of an explicitly hardened control, and `failed_redeem_attempts` exists precisely for this.
**Fix direction:** mirror the lockout inside the RPCs (per device id), or in the `staff-redemption` edge function before calling the RPC, and log failures.

**5. The `redemptions_owner_read` RLS policy is dead code: the migration revokes all table privileges on `redemptions` from `authenticated`, so owners cannot read their redemption audit trail via PostgREST.**
[migration :90](supabase/migrations/20260712120000_redemption_mode_staff_sessions.sql:90) `REVOKE ALL ON public.redemptions FROM anon, authenticated` removes the role-level SELECT grant; the policy at [:200-211](supabase/migrations/20260712120000_redemption_mode_staff_sessions.sql:200) can never matter because RLS policies filter rows but cannot substitute for a missing GRANT. No client or edge code currently reads `redemptions` (verified: the only references are inserts via the RPC), so nothing breaks *today* — but the rls-checklist doc (line 12) claims "owners can read redemption audit rows for their business," which the schema does not deliver, and the first owner-facing history screen will silently get `permission denied`. Either `GRANT SELECT ON public.redemptions TO authenticated` (the hash-bearing columns are not on this table, so it's safe with the policy) or drop the policy and route history through an edge function.

**6. Owner account deletion permanently bricks a locked counter device and orphans the staff Auth user.**
`delete-user-account` purges and deletes only the owner ([supabase/functions/delete-user-account/index.ts:79-116](supabase/functions/delete-user-account/index.ts:79)); `purge_user_data` (20260714120000) touches no redemption tables. The cascade chain: owner `auth.users` row → `businesses` → `redemption_devices` (business_id/owner_id CASCADE, [migration :14-15](supabase/migrations/20260712120000_redemption_mode_staff_sessions.sql:14)) → `redemptions`. But the *staff* Auth user is a separate `auth.users` row that nothing deletes — it survives forever with a valid refresh token and `app_role=redeemer` claims pointing at a vanished business (all guards then fail, so it can do nothing of consequence, but orphans accumulate). Worse for UX: the still-locked counter device can never exit — `exit-redemption-mode` returns 404 "Device is not active" once the device row is gone ([supabase/functions/exit-redemption-mode/index.ts:55-57](supabase/functions/exit-redemption-mode/index.ts:55)), and the client only force-clears when local state is missing ([lib/redemption-mode.ts:296-299](lib/redemption-mode.ts:296)); the device is stuck on the locked screen until app data is wiped.
**Fix direction:** delete-user-account should enumerate `redemption_devices.staff_user_id` for owned businesses and `auth.admin.deleteUser` each before deleting the owner; client exit should treat 404 as "device no longer exists → force-clear and log out."

**7. Deactivating a device does not revoke the staff user's sessions or refresh token — only the DB row flag.**
`manage-redemption-devices` `deactivate` flips `active=false` ([supabase/functions/manage-redemption-devices/index.ts:108-121](supabase/functions/manage-redemption-devices/index.ts:108)); there is no `auth.admin.signOut(staffUserId)` and the staff user keeps refreshing tokens indefinitely. **Mitigation is real and verified:** every consequential surface re-checks the device row server-side — `is_active_redeemer_for_business()` requires `active = true AND removed_at IS NULL AND deactivated_at IS NULL` ([migration :177-186](supabase/migrations/20260712120000_redemption_mode_staff_sessions.sql:177)), so the staff RPCs, the deals read guard, and the businesses read guard all die *immediately* on deactivation, not after token expiry. What the live token retains is the harmless authenticated-RPC surface from Finding 13 (aggregate claim counts, public geo reads), forever, until the device is `remove`d. `remove` does delete the staff user ([index.ts:127](supabase/functions/manage-redemption-devices/index.ts:127)), which kills the refresh token; in-flight access tokens (≤1h) still pass PostgREST signature checks after deletion but hit the same dead guards.
**Fix direction:** add `auth.admin.signOut(staffUserId)` on deactivate (cheap, closes the residual surface). Re-activation of an existing device similarly rotates the password ([activate-redemption-mode/index.ts:175-179](supabase/functions/activate-redemption-mode/index.ts:175)) without revoking prior sessions.

**8. The entire `redemptionMode.*` i18n namespace is missing from en/es/ko — every counter-screen and settings string ships English-only via inline `defaultValue`.**
Verified: `redemptionMode` appears in zero locale files ([lib/i18n/locales/en.json](lib/i18n/locales/en.json), es.json, ko.json — grep matches only the three component files). ~50 user-facing strings across [app/redemption-mode.tsx](app/redemption-mode.tsx), [app/(tabs)/redeem.tsx](app/(tabs)/redeem.tsx), and [components/redemption-mode-settings.tsx](components/redemption-mode-settings.tsx) rely on `defaultValue`. Additionally hardcoded with no `t()` at all: device status labels "Active"/"Inactive"/"Ready" ([components/redemption-mode-settings.tsx:36-40](components/redemption-mode-settings.tsx:36)), the default device label "Front Counter iPhone" ([:59](components/redemption-mode-settings.tsx:59)), and several thrown error strings in [lib/redemption-mode.ts:235,263,285](lib/redemption-mode.ts:235). Server error messages ("Incorrect exit PIN.", "Too many incorrect PIN attempts...") are English and rendered raw in banners. This is exactly the Batch 2b failure mode, on the screens a Spanish-speaking shop's staff stare at all day.

**9. The lock is enforced by effect-based redirects, so a cold start or deep link briefly mounts unlocked UI before the gate replaces the route.**
`RedemptionModeGate` is a `useEffect` redirect ([components/providers/redemption-mode-provider.tsx:98-105](components/providers/redemption-mode-provider.tsx:98)), mounted *after* five deep-link handlers in the tree ([app/_layout.tsx:87-93](app/_layout.tsx:87)). On cold start with a redeemer session, `tab-mode` sets mode "business" ([lib/tab-mode.tsx:85-89](lib/tab-mode.tsx:85)) and [app/index.tsx:66](app/index.tsx:66) redirects to `/(tabs)/create` before the gate's effect fires and replaces to `/redemption-mode`. A notification tap or share deep link can similarly land on a deal route for a frame. **Defense in depth holds:** every query those screens fire runs with the staff JWT and is RLS-blocked (block-all restrictive policies), and `isLocked` is belt-and-braces (`Boolean(state) || isRedeemerSession(session)`, [provider:75](components/providers/redemption-mode-provider.tsx:75)), so this is a flash of empty business chrome, not data exposure. Re-lock on cold start was verified: boot always re-reads AsyncStorage state and the session, and clearing app data destroys the staff session along with the lock (logged-out, not unlocked). Android back is captured ([app/redemption-mode.tsx:73-77](app/redemption-mode.tsx:73)) and the screen has `gestureEnabled: false` ([app/_layout.tsx:110](app/_layout.tsx:110)).

**10. Staff redemptions are invisible to the analytics pipeline the batch stack standardized on.**
`redeem-token` writes a `redeem_completed` row to `app_analytics_events` ([supabase/functions/redeem-token/index.ts:339-350](supabase/functions/redeem-token/index.ts:339)); `confirm_staff_redemption` writes only to `redemptions`. Dashboards built on `app_analytics_events` (and `failed_redeem_attempts` for fraud review) undercount once a shop adopts Redemption Mode. `deal_claims.redeemed_at`-based merchant insights stay correct because both paths set the same columns. Low urgency, but decide before pilot whether `redemptions` or `app_analytics_events` is the canonical count.

### LOW

**11. Exit-PIN failure counting is a non-atomic read-modify-write.**
[supabase/functions/exit-redemption-mode/index.ts:77-88](supabase/functions/exit-redemption-mode/index.ts:77) reads `pin_failed_attempts`, increments in JS, writes back. Parallel wrong-PIN requests can land the same `nextAttempts` value, so >5 attempts can fit before lockout. Same pattern in `owner-redemption-security` `failPin` ([index.ts:30-60](supabase/functions/owner-redemption-security/index.ts:30)). Practical impact is a few extra guesses against a 4–6 digit PIN; an atomic SQL `UPDATE ... SET pin_failed_attempts = pin_failed_attempts + 1 RETURNING` would close it.

**12. The staff-session SecureStore backup is mostly dead after refresh-token rotation.**
`STAFF_SESSION_BACKUP_KEY` stores the *original* session from activation ([lib/redemption-mode.ts:275](lib/redemption-mode.ts:275)). The live client session auto-refreshes and rotates its refresh token; the backup's token goes stale, so the recovery path at [lib/redemption-mode.ts:212-221](lib/redemption-mode.ts:212) will usually fail exactly when needed, dropping the device into the "expired/Owner PIN required" state. Functional papercut, not security (the device stays locked).

**13. Residual PostgREST/RPC surface of a staff token — enumerated and rated.**
Complete walk of everything `authenticated` can reach with a redeemer JWT (this is the Area 7 result; all verified against migrations):
- **Tables:** restrictive `redeemer_*_block_all` policies cover all 20 enumerated tables ([migration :279-317](supabase/migrations/20260712120000_redemption_mode_staff_sessions.sql:279)) — `deal_claims`, `deal_shares`, `push_tokens`, `consumer_profiles`, `profiles`, notification-pref columns, `favorites`, `failed_redeem_attempts`, etc. all return zero rows / denied. `deals` is readable only for the staff device's own *active* deals ([:214-226](supabase/migrations/20260712120000_redemption_mode_staff_sessions.sql:214)), `businesses` only its own row subject to the existing PII column grants ([:249-256](supabase/migrations/20260712120000_redemption_mode_staff_sessions.sql:249)); all writes blocked. `redemptions`/`redemption_devices`/`owner_redemption_security` have no authenticated grants at all. Hostile cross-tenant test (business A staff vs business B): every policy keys on `is_active_redeemer_for_business(<row's business_id>)`, which pins business_id AND device_id AND `staff_user_id = auth.uid()` from server-set JWT claims — no path to B's rows. **Pass.**
- **Storage:** read of `deal-photos`/`business-logos` is public anyway; writes require `businesses.owner_id = auth.uid()` ([20260706130000:14-21](supabase/migrations/20260706130000_deal_photo_owner_upload_policies.sql)), which a staff user never satisfies. **Pass.**
- **RPCs with authenticated EXECUTE** (full list from migration grep): `preview/confirm_staff_redemption` (intended), `validate_business_invite`/`report_business`/`report_user` (redeemer-rejected in this migration, [:632, :665, :700](supabase/migrations/20260712120000_redemption_mode_staff_sessions.sql:632)), `merchant_deal_insights`/`merchant_business_insights` (raise `forbidden` for non-owners), `ai_compose_quota_status` (owner check), `get_my_business()` (returns null), `deal_claim_visible_to_business_owner` (false for non-owners), `purge_user_data` (service-role only post-`fd1e98e`), and three that *do* answer a staff token: `deal_claim_counts(uuid[])` (aggregate counts for arbitrary deal ids — counts only, no rows), `nearby_businesses`/`nearby_deals`/`haversine_miles` (also granted to anon; public data; SECURITY INVOKER so the redeemer deals guard applies anyway), `lookup_deal_share` (anon-granted, public share metadata). **Coffee-shop standard met:** a stolen staff token can perform staff redemptions for its own business, read its own active deals, and read aggregates/public data — nothing else of consequence. The notable hygiene gap is Finding 3 (anon grants), not this surface.
- **Edge functions:** all 24 owner/consumer functions reject redeemer JWTs via the single shared `isRedeemerUser` guard (verified by grep — no function rolls its own check; `staff-redemption` uses the same module inverted). The four unguarded functions are auth-irrelevant: `billing-checkout-redirect` (static HTML redirect), `deal-link` (public), `cancel-visual-redeem` (auth-free stub), `stripe-webhook`/`weekly-deal-digest` (signature/cron-secret).
- **Caveat:** the block-all list is a frozen enumeration with `to_regclass` existence checks — any future table added to the schema is NOT automatically covered. Add "create the `redeemer_<table>_block_all` policy" to the new-table checklist.

**14. `exit-redemption-mode` as an unauthenticated public endpoint — audited, design holds, minor notes.**
([supabase/config.toml:413-416](supabase/config.toml:413) `verify_jwt = false`; no Authorization check in the function — intentional, the locked device may hold an expired session.)
- **Entropy/generation:** exit token is 32 bytes from `crypto.getRandomValues` ([_shared/redemption-crypto.ts:41-45](supabase/functions/_shared/redemption-crypto.ts:41)), stored client-side in SecureStore ([lib/redemption-mode.ts:276](lib/redemption-mode.ts:276)), stored server-side as SHA-256 ([crypto.ts:52-54](supabase/functions/_shared/redemption-crypto.ts:52)) with constant-time compare. 256-bit token + plain SHA-256 is correct for a high-entropy secret (KDF unnecessary).
- **Ordering is right:** lockout check → exit-token verify (401, no counter increment) → PIN verify (increments, locks at 5 for 5 min) ([index.ts:59-99](supabase/functions/exit-redemption-mode/index.ts:59)). An attacker *without* the exit token cannot touch the PIN counter, so the lockout cannot be used as a remote DoS lever — it scopes per-device-row and only token-holders reach it. Not per-IP, not global. **Correct scoping.**
- **What a correct guess yields:** `active=false` on the device row — i.e., the attacker can deactivate the shop's device (nuisance), nothing more; owner-session restore happens purely client-side from SecureStore. Replay of a captured request is a no-op re-deactivation (and a captured request already contains the PIN, so replay adds nothing).
- **Oracles:** 404 (no such/removed device) vs 401 (bad token) vs 403 (bad PIN) does distinguish device existence — but device ids are UUIDv4, unguessable. Timing distinguishes bad-token (1 SHA-256) from bad-PIN (120k-iteration PBKDF2), which only tells a token-holder what they already know. Acceptable.
- **Minor:** the endpoint accepts exits for already-inactive devices (idempotent, fine) and the exit token is not rotated after use (the row is dormant until re-activation regenerates both secrets — fine).

**15. PIN cryptography is sound — one notch below the requested bar but acceptable.**
PBKDF2-SHA256, 120k iterations, 16-byte random salt, fixed-length constant-time comparison ([_shared/redemption-crypto.ts:61-106](supabase/functions/_shared/redemption-crypto.ts:61)). Not bcrypt/scrypt/argon2, but PBKDF2@120k is a real KDF (OWASP floor for PBKDF2-SHA256 is 600k; for a 5-attempt-server-throttled 4–6 digit PIN the iteration count is not the binding constraint — the throttle is, see Finding 2 for the throttle bypass). Hashes live in `redemption_devices.pin_hash` / `owner_redemption_security.pin_hash`, both revoked from anon/authenticated ([migration :89-91](supabase/migrations/20260712120000_redemption_mode_staff_sessions.sql:89)) — readable by service role only; the management endpoints return safe shapes (verified: `list` selects no hash columns, `status` returns only booleans).
